Wrong Recipient — US Error Diagnostic

Answer each question to reach the correct handling outcome

Start
Customer mistake (MM) → NOT a TTS error

For Mobile Money: if the customer entered wrong details (wrong name, wrong number, wrong contact) but the name check confirms the correct recipient, we are not obligated to refund. We provided a warning, pre-disclosure page, and cancellation rights.

Exception — bank transfers: a wrong account number is always investigated even if the customer admits the mistake. See the red rule.

❌ Do NOT tick the Error field in ZD (MM only)

Names match → confirmed correct delivery

If "Name from provider" matches "Name entered by user" (100%, or 2/3, or 3/4 names), we can confirm the transfer reached the designated recipient. Error is not asserted.

MM only: do not tick the Error field. Bank transfers: always tick the Error field even when names match, because a wrong name is always initially recorded as a potential error for bank transfers.

❌ MM: Do NOT tick  |  ✅ Bank: Always tick

Cannot confirm delivery → error asserted

If names don't match, no name check was done, or the customer reports an issue we can't immediately dismiss (wrong account number, wrong bank, currency mismatch, closed account), we must investigate. Tier 2 refunds and attempts fund recovery.

✅ Tick the Error field in ZD

No name check done → always escalate

If "Name from provider" is empty, we cannot confirm or deny correct delivery. Always escalate to Tier 2 to request a Proof of Delivery (POD) from the partner. Do not close the case without a POD or 90-day SLA.

✅ Tick the Error field in ZD

Bank transfers only: Wrong account number or wrong digit count always requires a full Tier 2 investigation — even if names match, never close the case at Tier 1.
CPU transfers: The cashout agent will verify the recipient ID before paying out the funds, so wrong delivery is extremely rare. If the transfer hasn't been picked up yet, offer cancellation — it's not a US error.
